What Is This Project?
Imagine you are running a tech startup. You have multiple services вЂ” your main API, a database, a caching layer, a background job processor. One day, users start complaining that the app is slow. You have no idea why. You check logs manually, but there are thousands of lines. You don't know which service failed first or what caused the chain reaction.
AI Incident Analyzer solves exactly this problem. It is an intelligent observability platform that:

Automatically detects when something goes wrong (anomaly detection)
Creates an "incident" вЂ” a record of the failure event
Uses AI (Gemini/Groq) to analyze the failure and explain: what happened, why it happened, and how to fix it
Real-world analogy: Think of it as a 24/7 DevOps engineer who watches all your services, never sleeps, and instantly tells you "Redis latency spiked в†’ caused API timeouts в†’ triggered worker retry storms" the moment something breaks.

Technology Choices вЂ” Why Each Tool
Every technology in this project was chosen for a reason. Here is a plain-English explanation:
Technology
Why We Use It
FastAPI
Modern Python web framework. Async-first, auto-generates API docs, extremely fast. Better than Flask for this use case because it natively supports WebSockets and async DB queries.
PostgreSQL
Relational database. We need ACID transactions for incident records. Structured data with relationships (service в†’ logs в†’ incidents). Much better than MongoDB here.
SQLAlchemy (async)
Python ORM to interact with PostgreSQL. We use the async version so database queries don't block the event loop вЂ” critical for a real-time system.
Alembic
Database migration tool. When you change your database schema (add a column, etc.), Alembic tracks and applies changes without losing data.
Redis
In-memory data store. Used for two things: (1) Pub/Sub messaging so WebSocket workers receive new log events, (2) Caching and rate limiting.
Celery
Distributed task queue. When we need to run heavy work (AI analysis, anomaly detection) without blocking the API, Celery handles it in the background.
WebSockets
Persistent two-way connection. Unlike HTTP (request в†’ response), WebSockets allow the server to push data to the browser the instant something happens.
Next.js
React framework for the frontend dashboard. File-based routing, built-in optimizations, easy deployment on Vercel.
TailwindCSS
Utility-first CSS. Build modern UIs rapidly without writing custom CSS files.
Zustand
Lightweight state management for React. Simpler than Redux, perfect for managing live log streams and incident state.
Docker
Containerization. Your app runs identically on your laptop and in production. No 'works on my machine' problems.
Gemini/Groq API
AI language models. Groq is extremely fast (inference speed). Gemini has a generous free tier. We use them to generate incident explanations.

Why This Separation Matters
This architecture follows the "Clean Architecture" or "Layered Architecture" principle. Here is the rule:
рџЏ—пёЏ Architectural Layers вЂ” Who Calls Who
routes/ (HTTP layer)   в†’   calls   в†’   services/ (business logic)
services/ (logic)      в†’   calls   в†’   repositories/ (data access)
repositories/          в†’   calls   в†’   models/ (database tables)
вљ пёЏ  NEVER skip layers. Routes should NOT directly query the database.
вљ пёЏ  Services should NOT know about HTTP requests or responses.
вљ пёЏ  Repositories should ONLY contain database queries, nothing else.
WHY? в†’ Each layer can be tested, swapped, or scaled independently.

Step-by-Step Breakdown
Step 2.1 вЂ” Service Registration
Before a service can send logs, it must register. Registration creates a service record with a unique API key. Think of it like creating an account.
POST /services/register accepts: { name, description, environment (prod/staging/dev) }
Returns: { service_id, api_key } вЂ” the service stores this API key and uses it for all future requests
Step 2.2 вЂ” API Key Authentication
Every log ingestion request must include the API key in the X-API-Key header. We verify it against the database and attach the service info to the request.
рџ”‘ Authentication Flow
1. Service sends: POST /logs with header X-API-Key: sk_prod_abc123
2. FastAPI dependency: get_current_service(api_key)
3. We query PostgreSQL: SELECT * FROM services WHERE api_key = ?
4. If not found в†’ 401 Unauthorized
5. If found but inactive в†’ 403 Forbidden
6. If valid в†’ inject service object into route handler
FastAPI Dependencies are perfect for this вЂ” reusable auth across all protected routes
Step 2.3 вЂ” Log Schema Design
Each log entry must be strictly validated. Here is what a log looks like:
рџ“‹ Log Payload Schema
{
  service_name: str           в†ђ Which service is sending this
  log_level: enum             в†ђ DEBUG / INFO / WARNING / ERROR / CRITICAL
  message: str                в†ђ The log message (max 10,000 chars)
  timestamp: datetime         в†ђ When it happened (ISO 8601 format)
  metadata: dict (optional)   в†ђ Extra context: { request_id, user_id, latency_ms }
  trace_id: str (optional)    в†ђ For distributed tracing correlation
}
Step 2.4 вЂ” Rate Limiting
Use Redis to implement a sliding window rate limiter. Each service is allowed 1000 log requests per minute. If exceeded, return 429 Too Many Requests.
How it works: Redis ZADD + ZCOUNT with timestamps. For each request, add current timestamp to a sorted set. Count entries in the last 60 seconds. If count > 1000, reject.
Step 2.5 вЂ” Heartbeat Tracking
Each service should send a POST /services/{id}/heartbeat every 30 seconds. This updates last_seen_at in the database. A background job (Phase 4) will detect services that stop sending heartbeats and mark them as down.

Key Concepts Explained
Why WebSockets Instead of HTTP Polling?
вљЎ WebSockets vs Polling
HTTP Polling:   Browser asks 'any new logs?' every 2 seconds в†’ wasteful, 500ms delay minimum
WebSockets:     Server pushes 'new log!' the instant it arrives в†’ 0ms delay, efficient
WebSocket lifecycle:
  Client в†’ server: 'I want to subscribe to logs for service payment-api'
  Connection stays OPEN (unlike HTTP which closes after each request)
  Server в†’ client: pushes log events as they arrive
  Either side can close the connection at any time
Redis Pub/Sub Architecture
Here is the challenge: we might run 3 FastAPI instances for load balancing. If a log arrives at instance #1, how do instances #2 and #3 know to send it to their WebSocket clients?
Answer: Redis Pub/Sub. When any FastAPI instance receives a log, it publishes to a Redis channel. All instances subscribe to that channel. All of them receive the message and can push to their clients.
рџ“Ў Redis Pub/Sub Flow
PUBLISHER (log ingestion endpoint):
  redis.publish('logs:payment-api', json.dumps(log_data))
SUBSCRIBER (WebSocket worker):
  async for message in redis.subscribe('logs:payment-api'):
      await websocket.send_json(message)
Result: All connected dashboard clients receive the log in <50ms
Channels used:
  logs:{service_name}   в†ђ logs from a specific service
  logs:all              в†ђ all logs across all services
  incidents:new         в†ђ new incident alerts
Connection Manager
We need a class that tracks all active WebSocket connections. When a new log arrives via Redis, we broadcast to all relevant connections. When a client disconnects (closes browser), we remove them from the list.

Key Concepts Explained
Sliding Window Algorithm
A sliding window counts events within a rolling time frame. Unlike a fixed window (counting resets every minute), a sliding window gives a more accurate view of recent activity.
рџ“Љ Sliding Window Example
Threshold: 10 errors in 60 seconds
FIXED WINDOW (bad):   
  12:00:55 в†’ 9 errors (no alert)
  12:01:00 в†’ window resets в†’ 0 errors (missed the 15 errors spanning the boundary!)
SLIDING WINDOW (good):
  12:01:00 в†’ count errors from 12:00:00 to 12:01:00 = 15 errors в†’ ALERT
Implementation: Use PostgreSQL query with WHERE timestamp > NOW() - INTERVAL '60 seconds'
Or use Redis sorted sets (ZADD timestamp, ZCOUNT from..to) for faster counting
Incident States
рџљ¦ Incident Lifecycle
OPEN      в†ђ Just detected, AI analysis pending
ANALYZING в†ђ AI is currently generating root cause analysis
RESOLVED  в†ђ Error rate dropped back below threshold
IGNORED   в†ђ Manually dismissed by engineer
State transitions are logged with timestamps for audit trail
Celery Worker Design
We use Celery Beat (a scheduler) to run the anomaly_detection task every 30 seconds. When an incident is detected:
Create incident record in PostgreSQL (status: OPEN)
Publish incident event to Redis в†’ broadcast to WebSocket clients
Queue a separate Celery task: run_ai_analysis(incident_id)
Why Celery instead of running detection in the API? The API must respond quickly (<100ms). Anomaly detection scans thousands of log records. This would make the API slow. Celery runs it in a separate process entirely.

рџђі Multi-Stage Dockerfile Explained
# Stage 1: Builder вЂ” install dependencies
FROM python:3.11-slim AS builder
COPY requirements.txt .
RUN pip install --user -r requirements.txt
# Stage 2: Runner вЂ” copy only what's needed
FROM python:3.11-slim AS runner
COPY --from=builder /root/.local /root/.local
COPY ./app ./app
CMD ['uvicorn', 'app.main:app', '--host', '0.0.0.0', '--port', '8000']
WHY multi-stage?  Build tools (gcc, pip cache) stay in Stage 1.
Final image only has runtime code вЂ” smaller, faster, more secure.
Typical result: 1.2GB в†’ 180MB image size

PHASE 9
Advanced Features
Vector Search В· Semantic Similarity В· Distributed Tracing В· AI Auto-Grouping
What Are We Building in Phase 9?
Phase 9 is for those who want to push further. These features transform the platform from a good portfolio project into something that matches real enterprise observability tools.
Feature 9.1 вЂ” Semantic Log Search with ChromaDB
Instead of searching logs by keyword ("redis error"), we embed log messages as vectors and search by meaning. "database connection refused" also finds "cannot connect to postgres" even though the words are different.
Use sentence-transformers to embed each log message
Store embeddings in ChromaDB (local vector database)
GET /logs/search?q=redis+timeout uses vector similarity search
Feature 9.2 вЂ” AI Incident Grouping
When multiple incidents are related (e.g., database is down в†’ API fails в†’ workers fail), AI groups them into a single "outage event" instead of showing 15 separate incidents.
Embed incident summaries as vectors
Cluster similar incidents using cosine similarity
AI generates a unified outage report for the cluster
Feature 9.3 вЂ” Service Dependency Graph
Visualize which services call which. When service A goes down, the graph shows which downstream services will be affected. Built from log correlation (trace_id matching).
Feature 9.4 вЂ” Distributed Tracing
Each request gets a trace_id. As it passes through multiple services, each logs with that trace_id. You can then see the full journey of a single request across services вЂ” exactly like Jaeger or Zipkin.
Feature 9.5 вЂ” Auto-Remediation
For known failure patterns (e.g., Redis memory full), the AI suggests a specific fix command. An authorized operator can click "Apply Fix" which triggers a Celery task that runs the remediation script.
вљ пёЏ Production Safety Note
Auto-remediation is powerful but dangerous.
Always require manual confirmation before executing commands.
Log every remediation action with who approved it and when.
Start with read-only fixes (clearing cache) before write operations.
This feature is 100% optional and only for advanced builders.

Common Mistakes to Avoid
вќЊ Mistakes Every Beginner Makes (and How to Avoid Them)
1. MISTAKE: Skipping Pydantic validation вЂ” 'I'll validate later'
   FIX: Always define request/response schemas FIRST before writing route logic
2. MISTAKE: Using synchronous SQLAlchemy in an async FastAPI app
   FIX: Use 'from sqlalchemy.ext.asyncio import AsyncSession' everywhere
3. MISTAKE: Storing credentials in code or .env files committed to git
   FIX: Add .env to .gitignore on day 1. Use GitHub secrets for CI/CD.
4. MISTAKE: Routes directly querying the database
   FIX: Routes в†’ Services в†’ Repositories в†’ Database. Always follow layers.
5. MISTAKE: Not handling WebSocket disconnects в†’ memory leak over time
   FIX: Always use try/finally in WebSocket handlers to remove from manager
6. MISTAKE: Blocking the async event loop with slow synchronous code
   FIX: Any operation >10ms should be in a Celery task or run_in_executor
7. MISTAKE: Not writing Alembic migrations вЂ” manually editing DB schema
   FIX: Every schema change must have a migration. Never ALTER TABLE manually.
